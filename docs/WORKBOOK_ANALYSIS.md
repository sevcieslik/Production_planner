# Workbook analysis: `sample-data/2PROD Capacity plan 2026.xlsx`

Audit date: 2026-06-19.

This is an audit-only analysis of the real capacity planning workbook. No app logic, schema, UI, or importers were changed. The repository workbook analyser entry point could not be executed in this container because the Python dependencies declared in `requirements.txt` were not installed and package installation was blocked by the environment. To complete the audit, the `.xlsx` package was inspected directly with Python standard-library ZIP/XML parsing.

## Worksheet details

### `Projects`

- **Dimensions:** `A1:Y223`.
- **Likely purpose:** Master project register with KPI fields, OSR/progress lookups, original estimates, team-manager end-date estimates, live project time tracking, and archive flags.
- **Key headers:** `Project Code`, `Project Name`, `Production % Complete`, `Lead`, `Circuit Length (km)`, `ROW (km)`, `Area (km2)`, `SPU Factor`, `Total SPUs`, `Bid Total`, `RS Total`, `GIS Total`, `PLS Total`, `Production Total`, `Production Start date`, `Production Estimated Completion Date`, `RS (Luke)`, `GIS (Dom)`, `PLS (Carlos)`, `Staffology Time Entries`, `Remaining Hours`, `Variance`, `SPUs Delivered`, `PM end date`, `Archived`.
- **Date ranges:** Numeric Excel date serials resolve to 2023-01-30 through 2026-03-23, primarily in start/completion/end-date columns.
- **Formulas/calculated areas:** Heavy lookup formulas against external Google Sheets/OSR-like sources (`IMPORTRANGE`, `VLOOKUP`, `IFERROR`, `IFNA`), totals such as `Production Total = SUM(RS/GIS/PLS)`, time-entry aggregation via `SUMIF`, delivered SPU lookups, and discipline-specific manager end dates via `LET`, `COUNTIFS`, and max-date logic.
- **Classification:** Mixed source/reporting. Project identity, estimates, archive status, and PM end dates are source-like; progress, actuals, totals, remaining hours, variance, and manager end-date fields are calculated/reporting.

### `Roster Daily`

- **Dimensions:** `A1:YI63`.
- **Likely purpose:** Person roster and daily availability calendar. Rows are staff members; columns after the role fields are weekday dates; cell values are available hours for that person on that day.
- **Key headers:** `Name`, `Team`, `Primary Role`, `Secondary Role`, followed by generated weekday date columns beginning at Excel serial `45838` (2025-06-30). Teams use discipline-coded labels such as `1_RS`, `2_GIS`, and `3_PLS`.
- **Date ranges:** The first visible date is 2025-06-30. Later date headers are generated by formula across the sheet; the sheet extends to column `YI`, indicating a long daily planning horizon.
- **Formulas/calculated areas:** Date header generation uses `WORKDAY.INTL` from the first date; many copied header formulas are present. Body cells are mostly entered availability hours such as `8`, `7.5`, `0`, or blanks.
- **Classification:** Source data. It maps directly to staff, teams, roles, normal daily capacity, exceptions, and availability calendar records.

### `Allocation Daily`

- **Dimensions:** `A1:YG60`.
- **Likely purpose:** Daily person-to-project allocation matrix. Rows are staff members; columns are dates; cell values are project names or non-project statuses such as `Training` and `Unavailable`.
- **Key headers:** `Name`, `Team`, followed by weekday date columns from 2025-06-30 to 2026-03-20 in the populated range.
- **Date ranges:** Detected date serial range is 2025-06-30 through 2026-03-20.
- **Formulas/calculated areas:** Date headers are generated for later columns using `WORKDAY.INTL`; assignment cells are largely manual labels. There are 465 formula cells, concentrated in generated header/date areas rather than in person assignments.
- **Classification:** Source data for daily allocations, with source-like non-billable/availability labels embedded in the same matrix.

### `Teams Breakdown`

- **Dimensions:** `A1:EF1022`.
- **Likely purpose:** Weekly calculation table that transforms daily roster hours and daily allocation labels into weekly allocated hours by project and discipline.
- **Key headers:** `Active`, `Project`, `Discipline`, `Planned HRS`, `HRS Assigned`, `HRS remaining`, followed by weekly date columns beginning 2025-06-30.
- **Date ranges:** Weekly date headers run from 2025-06-30 through 2027-12-20.
- **Formulas/calculated areas:** Dense calculation sheet. Weekly cells use `SUMPRODUCT`, `FILTER`, `INDIRECT`, and references into `Roster Daily` and `Allocation Daily` to sum matching person availability hours where daily allocation labels equal the project and person team equals the discipline. A multiplier of `0.85` appears in the weekly allocation formulas. Totals use `SUM` and `SUMIF` to compute assigned and remaining hours.
- **Classification:** Calculation/intermediate. It should not be treated as a primary source if the app can derive the same values from roster and allocation records.

### `Capacity`

- **Dimensions:** `A1:CJ1069`.
- **Likely purpose:** Project-level capacity dashboard showing progress, planned hours, assigned hours, remaining hours, effort-spent percentage, and weekly project allocation roll-ups.
- **Key headers:** `Project code`, `Project name`, `% completed`, `HRS Assigned`, `Planned HRS`, `HRS remaining`, `% of effort spent`, followed by weekly date columns beginning 2025-06-30.
- **Date ranges:** Weekly date headers run from 2025-06-30 through 2027-01-11.
- **Formulas/calculated areas:** Pulls `% completed` from an external `Production` range, planned hours from `Projects`, assigned weekly hours from `Teams Breakdown` via `SUMIF`, total assigned via `SUM`, future remaining via `SUMIF` against current week, and effort-spent percentage from assigned vs remaining/planned formulas.
- **Classification:** Reporting/dashboard. It aggregates `Projects` and `Teams Breakdown`; it is not the preferred source for normalized imports except as a validation target.

## Gaps between workbook and current MVP

1. **Direct workbook import is not implemented.** The MVP has an analyser/profile utility, but controlled imports currently focus on CSV/XLSX schemas for projects, OSR progress, and holidays rather than this matrix-shaped workbook.
2. **People/roster import is not implemented end-to-end.** The workbook has rich staff roster, team, primary role, secondary role, and daily availability data. The schema can store people and availability, but importer support is still only declared for future work.
3. **Daily allocation matrix import is not implemented.** The MVP supports manual daily allocations, but it cannot yet ingest `Allocation Daily` as a wide date-column matrix and normalize it into `daily_allocations`.
4. **Project budget extraction from `Projects` is not implemented.** The workbook's RS/GIS/PLS estimate columns map well to `project_discipline_budgets`, but no workbook-specific importer exists yet.
5. **Workbook-specific non-billable labels need controlled mapping.** Labels such as `Administrative`, `Training`, and `Unavailable` are used as allocation values and calculation buckets. The MVP has `non_billable_categories`, but it needs a mapping policy before import.
6. **Discipline manager end-date fields are richer than current project dates.** Columns `RS (Luke)`, `GIS (Dom)`, and `PLS (Carlos)` imply discipline-level forecast completion dates. The MVP has project schedules but no explicit discipline-specific completion-date entity.
7. **SPU/productivity metrics are not represented.** Circuit length, ROW, area, SPU factor, total SPUs, and SPUs delivered are absent from the current normalized schema.
8. **The workbook derives assigned hours with a 0.85 factor.** The MVP should document and decide whether that utilization/efficiency factor is a business rule before attempting workbook parity.
9. **External Google Sheets/OSR dependencies are embedded in formulas.** Many project/progress fields are formulas against external `IMPORTRANGE` and broken `#REF!` ranges. A reliable app import should prefer explicit source exports or APIs over evaluating spreadsheet formulas.
10. **Wide daily/weekly matrices need normalization and validation.** The workbook stores dates as columns; the app stores normalized rows. Import must validate dates, people, projects, non-billable labels, and duplicates before writing.

## Current MVP features not represented in the workbook but should remain

1. **Audit log.** The workbook does not provide a durable audit trail of imports and edits; the app's `audit_log` remains important.
2. **Role-based UI behavior.** The workbook does not model CDO/Manager/PM permissions; app roles should remain.
3. **Alerts.** The workbook has calculated variances but not persistent alert records for unfilled demand, underallocation, overallocations, stale imports, or deadline risk.
4. **Forecast snapshots.** Workbook formulas show current-state calculations; the app's `forecasts` table can keep dated forecast records for trend analysis.
5. **Controlled imports and exports.** The workbook is a free-form operational artifact; the MVP's constrained importer/exporter pattern should remain for data quality.
6. **Normalized non-billable categories.** Workbook labels are embedded in assignment cells; app categories are a cleaner structure and should remain.
7. **Project schedules/milestones.** The workbook has date fields but no generalized milestone table; the MVP's schedule model remains useful.
8. **Local-first SQLite structure.** The workbook is not a replacement for normalized storage, validation, and traceability.

## Recommended next implementation step

Implement a **read-only workbook staging importer** that parses the five known sheets and produces validation previews without writing to production tables by default. The first slice should normalize and preview:

1. `Projects` rows into candidate `projects` and `project_discipline_budgets` records.
2. `Roster Daily` rows into candidate `people`, `teams`/`disciplines`, and `availability_calendar` records.
3. `Allocation Daily` rows into candidate `daily_allocations`, with unresolved project/category labels reported as validation errors.

Do not start with `Teams Breakdown` or `Capacity` as import sources. Treat those as reconciliation targets: after staging projects, roster, and allocations, derive equivalent weekly summaries in the app and compare them against the workbook's `Teams Breakdown`/`Capacity` totals.
